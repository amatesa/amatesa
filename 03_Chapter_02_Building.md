# Chapter 02 — Building

Understanding a problem is only the beginning.

Sooner or later, I need to build something.

Sometimes it's a report that helps a manager see a pattern that wasn't obvious before.

Sometimes it's documentation that allows a new team member to become productive faster.

Sometimes it's a spreadsheet that removes repetitive work.

Sometimes it's a dashboard.

Sometimes it's a website.

Sometimes it's a game.

The final product changes.

The intention behind it doesn't.

I build because building is how ideas become useful.

Throughout my career I noticed something about myself.

Whenever I repeat the same task for too long, my curiosity starts looking elsewhere.

Not because I lose interest in the work.

Because I begin asking whether the work itself could be improved.

Can this process be simplified?

Can this information be organized differently?

Can someone else understand this faster?

Can repetitive work be reduced?

Can this experience become clearer?

Those questions usually lead me to create something.

Sometimes that "something" is visible.

A website.

A prototype.

A dashboard.

A playable game.

Other times it is almost invisible.

A procedure.

A document.

A workflow.

A naming convention.

A better way to organize information.

Many of the things I'm most proud of were never official projects.

Nobody asked me to create them.

They appeared because solving one problem revealed another opportunity to improve the system.

Looking back, I realize that I rarely think in terms of isolated tasks.

I naturally think in systems.

Every task belongs to a larger process.

Every process belongs to a larger workflow.

Every workflow affects people.

Understanding those relationships makes building much easier.

It also changes what "building" means.

Building is not limited to writing code.

Building means creating something that did not exist before and leaves the situation better than it was.

That idea has followed me from healthcare to fintech, from operations to data analysis, from web development to game development.

The medium changes.

The objective remains remarkably consistent.

I don't build because I enjoy technology.

I build because I enjoy making things work better.

If tomorrow I needed to learn a completely different discipline to solve a meaningful problem, I would approach it the same way I always have.

Understand.

Learn.

Build.

Improve.

Then repeat.

That's probably the most consistent pattern throughout my entire career.

Not the industries.

Not the tools.

The process.
