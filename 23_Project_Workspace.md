# Screen 23 — Project Workspace

## Purpose

The Project Workspace is the core experience of the portfolio.

It transforms every project into an interactive case study.

Visitors should feel they are opening Alejandro's working environment rather than reading documentation.

Each project becomes a temporary workspace that exists inside the portfolio.

When closed, it returns naturally to the Build environment.

---

# Philosophy

Projects are evidence.

Not products.

Not achievements.

Not marketing pages.

Every project demonstrates a way of thinking.

Technology supports the story.

It never becomes the story.

---

# User Goal

Visitors should answer one question after every project.

"Could I trust this person to solve a problem?"

If the answer is yes,

the workspace has succeeded.

---

# Navigation

Opening a project never feels like leaving the portfolio.

The Build workspace remains alive in the background.

The camera simply approaches one specific project.

Closing the project returns to the exact previous position.

The visitor never loses orientation.

---

---

# Section 01 — Introduction

Purpose

Present the project in one minute.

Content

Title

Short summary

Role

Timeline

Project type

Status

Duration

Links

The visitor immediately understands what the project is.

---

# Section 02 — Problem

Purpose

Describe the challenge.

Never describe features.

Instead explain:

What problem existed?

Who experienced it?

Why was it worth solving?

Success begins with understanding.

---

# Section 03 — Context

Purpose

Provide enough background to understand decisions.

Context may include:

Business constraints

Academic requirements

Technical limitations

Available resources

Team size

Deadlines

The visitor understands why certain decisions were made.

---

# Section 04 — Investigation

Purpose

Reveal how understanding was built.

Possible evidence

Research

Sketches

Notes

Documentation

References

Experiments

Failed ideas

Visitors should see thinking.

Not only outcomes.

---

# Section 05 — Approach

Purpose

Explain the reasoning.

Questions answered

Why this solution?

Why not another?

What alternatives existed?

What assumptions were tested?

Reasoning is more valuable than implementation.

---

# Section 06 — Iterations

Purpose

Demonstrate improvement.

Show evolution.

Not perfection.

Possible elements

Version comparisons

Before / After

Prototype progression

Feedback

Changes

Learning

Visitors should understand refinement.

---

# Section 07 — Solution

Purpose

Present the final implementation.

This section may contain

Screenshots

Videos

Interactive demos

Architecture

Results

Documentation

Repositories

Technology supports understanding.

---

# Section 08 — Reflection

Purpose

Describe what changed after finishing.

Suggested questions

What surprised me?

What would I improve?

What would I build differently today?

What did I learn?

Reflection demonstrates maturity.

---

# Section 09 — Technology

Technology intentionally appears near the end.

Suggested format

Languages

Frameworks

Libraries

Tools

Platforms

Development Environment

Version Control

Deployment

Technology explains implementation.

It never justifies the project.

---

# Section 10 — Resources

Possible items

GitHub

Documentation

Playable Demo

Video

PDF

Downloads

External Links

Everything optional.

Nothing mandatory.

---

# Workspace Layout

Desktop

Two-column layout.

Left

Narrative.

Right

Media.

Media remains synchronized with the current section.

Images change naturally while scrolling.

---

# Sticky Media

Media remains visible.

Scrolling changes the explanation.

Not the layout.

Visitors always understand what they are reading.

---

# Three.js Integration

The workspace exists inside the larger environment.

When opening a project,

the surrounding workspace fades softly.

The selected project gains focus.

When closing,

the larger system returns.

Three.js reinforces continuity.

Never interruption.

---

# Motion

Sections slide naturally.

Images transition softly.

Camera movement remains subtle.

The workspace feels calm.

Every transition communicates progression.

---

# Mobile

Narrative first.

Media second.

Sticky behavior disabled.

Everything becomes linear.

Reading remains effortless.

---

# Accessibility

Every image requires descriptive alt text.

Videos include captions.

Keyboard navigation reaches every section.

Media controls remain fully accessible.

The workspace functions without JavaScript.

---

# Performance

Images lazy loaded.

Videos loaded on demand.

Three.js paused when inactive.

Heavy media deferred.

Text always loads first.

---

# Content Rules

Every project answers five questions.

Why did it exist?

How was it understood?

How was it solved?

What changed?

What was learned?

If any question is missing,

the project is incomplete.

---
