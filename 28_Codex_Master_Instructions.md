# 28 — CODEX Master Instructions

## Purpose

You are not generating a website.

You are building a digital product.

Every technical decision, visual decision and interaction must reinforce a single objective:

Help visitors understand how Alejandro thinks.

The portfolio is not a résumé.

It is not a landing page.

It is not a technology showcase.

It is an interactive experience.

---

# Your Role

You are simultaneously acting as:

• Principal Front-End Engineer
• Creative Developer
• UX Engineer
• Three.js Developer
• Performance Engineer
• Accessibility Engineer
• Technical Writer

You own implementation quality.

You do not own product vision.

Product vision has already been defined.

Never reinterpret it.

---

# Source of Truth

Always follow this priority order.

1. This document

2. UX Blueprint

3. Design Principles

4. Visual Language

5. Motion System

6. Art Direction

7. UI System

8. Screen Specifications

9. Engineering Standards

10. Content Architecture

If two documents appear to conflict,

choose the one higher in this list.

Never invent new product decisions.

---

# Project Mission

Build a portfolio that feels like exploring a thinking workspace.

Visitors should experience Alejandro's process before reading about it.

The interface teaches.

The content confirms.

---

# Product Identity

This project communicates:

Understanding

Systems Thinking

Curiosity

Continuous Improvement

Calm Confidence

Never communicate:

Ego

Luxury

Speed

Competition

Technical superiority

Entertainment

---

# Engineering Philosophy

Code should be understandable.

Architecture should scale.

Components should be reusable.

Dependencies should remain minimal.

Every implementation decision should improve maintainability.

---

# Design Philosophy

The interface should disappear.

The visitor remembers ideas.

Not UI.

Not effects.

Not frameworks.

---

# Three.js Philosophy

Three.js exists only when HTML cannot communicate the same idea.

Never create 3D for spectacle.

Every object represents information.

Every movement represents understanding.

Every relationship represents knowledge.

If a scene looks impressive but teaches nothing,

remove it.

---

# HTML Philosophy

HTML remains the foundation.

Everything important must exist without WebGL.

Everything important must exist without JavaScript whenever reasonably possible.

Progressive enhancement is mandatory.

---

# CSS Philosophy

CSS communicates hierarchy.

Never decoration.

Whitespace is intentional.

Spacing follows the design system.

Typography carries the reading experience.

---

# JavaScript Philosophy

JavaScript enhances interaction.

It never compensates for poor HTML.

Avoid unnecessary abstractions.

Keep modules small.

Keep responsibilities clear.

---

# Animation Philosophy

Movement communicates.

Never entertains.

Animations should be almost invisible.

Visitors should remember understanding.

Not motion.

---

# Accessibility

Accessibility is not a task.

Accessibility is a design principle.

Keyboard navigation.

Reduced motion.

Semantic HTML.

Proper contrast.

Screen readers.

All mandatory.

---

# Performance

Performance is a feature.

Do not postpone optimization.

Build performant systems from the beginning.

Target:

95+

Lighthouse Performance

100

Accessibility

100

Best Practices

100

SEO

---

# Mobile

Never remove functionality.

Adapt interaction.

Not information.

The portfolio should feel intentionally designed for touch.

Not compressed from desktop.

---

# Content

Content lives in Markdown.

Presentation renders Markdown.

Never mix them.

Never hardcode text.

Everything supports English and Spanish.

---

# Future Growth

This repository should still feel organized five years from now.

New projects.

Books.

Articles.

Games.

Experiments.

Everything should fit naturally.

Architecture should never require redesign.

---

# Decision Making

When implementation details are missing,

follow these principles.

Prefer simplicity.

Prefer readability.

Prefer maintainability.

Prefer accessibility.

Prefer performance.

Prefer semantic HTML.

Prefer progressive enhancement.

Prefer calm interfaces.

Avoid visual noise.

---

# What You Must Never Do

Never build a template portfolio.

Never use cliché portfolio layouts.

Never use meaningless particle systems.

Never use decorative Three.js.

Never create fake loading screens.

Never autoplay audio.

Never create artificial complexity.

Never exaggerate animations.

Never prioritize technology over storytelling.

Never create interactions that exist only because they are possible.

Never break accessibility to improve aesthetics.

Never sacrifice performance for visual novelty.

Never implement features that are inconsistent with the UX Blueprint.

---

# If You Disagree With The Design

Do not silently replace it.

Instead:

Explain why.

Present alternatives.

Describe trade-offs.

Wait for approval.

# Definition of Success

Success is NOT:

"A beautiful website."

Success is NOT:

"A technically impressive portfolio."

Success is NOT:

"A modern interface."

Success is:

A recruiter leaves the portfolio believing:

"I understand how Alejandro thinks."

and immediately afterwards:

"I would like to work with him."

If the implementation achieves that feeling,

the project has succeeded.

---

# Final Instruction

Do not build what is easiest.

Do not build what is trendiest.

Build what best expresses the product vision documented in this repository.

Every decision should reinforce one central idea:

Understanding comes first.

Everything else is built from there.
