# Screen 21 — Think

## Purpose

Think is the first chapter of the portfolio.

It does not explain Alejandro's mindset.

It demonstrates it.

Visitors should experience how ideas become understanding before reading a single paragraph.

This chapter establishes the mental model used throughout the rest of the portfolio.

Everything that follows is a consequence of what happens here.

---

# UX Goal

Visitors should unconsciously understand one idea:

Understanding comes before building.

The interface should never tell them this.

It should allow them to experience it.

---

# Emotional Goal

Beginning

Curiosity

↓

↓

Pattern Recognition

↓

Understanding

↓

---

# Narrative

The Home screen asked:

"Who is this?"

Think answers:

"This is how he approaches complexity."

The visitor should leave this chapter believing:

"He doesn't jump to solutions."

"He first tries to understand the system."

---

# Scene Composition

Unlike the Hero, this chapter becomes more immersive.

Whitespace decreases slightly.

The Three.js scene becomes the main visual language.

Text becomes secondary.

The interface transitions from presentation to exploration.

---

# Three.js Scene

## Narrative Concept

The visitor enters a three-dimensional conceptual workspace.

Nothing represents technology.

Nothing represents computers.

Everything represents ideas.

---

## Initial State

The scene contains isolated nodes.

They are not connected.

Each node represents an observation.

Examples:

Problem

User

Data

Constraint

Idea

Process

Feedback

Goal

The visitor doesn't know this yet.

---

## Behaviour

Nodes slowly move.

Not randomly.

They search for equilibrium.

The movement feels intelligent.

Not procedural.

---

## Mouse Interaction

When the cursor approaches a node,

the node reacts.

Nearby nodes begin responding.

Some connections appear.

Others disappear.

The visitor realizes:

Changing one idea affects the entire system.

---

## Discovery

After interacting with several nodes,

relationships begin emerging.

Clusters appear.

Patterns become visible.

The scene slowly becomes understandable.

Nothing is explained.

Understanding is earned.

---

# Camera

The camera behaves like attention.

When the visitor focuses on something,

the camera subtly approaches.

When the visitor continues scrolling,

the camera widens again.

The visitor never loses orientation.

# Transition to Build

This transition is critical.

When scrolling,

individual nodes begin forming larger structures.

The visitor realizes:

Understanding naturally becomes construction.

The Three.js scene transforms continuously.

No hard cut.

No page transition.

The camera slowly approaches one of these structures.

That structure becomes the first project inside Build.

The visitor should feel:

"I didn't leave this chapter."

"I naturally arrived at the next one."

---

# Accessibility

The editorial content exists independently from the interaction.

Visitors using screen readers receive the same ideas.

Reduced motion replaces animated relationships with static diagrams.

---

# Performance

Maximum GPU budget should remain significantly below the Hero.

Connections should be generated efficiently.

Avoid particle systems.

Avoid unnecessary shaders.

Optimize for integrated GPUs.

---

# Comments for CODEX

Do not build a network visualization.

Build a conceptual thinking model.

Every visual decision should communicate that understanding emerges from relationships.

The visitor should finish this chapter believing that Alejandro first observes, then connects, and only afterwards builds solutions.

If users describe this chapter as "a cool graph", it has failed.

If they describe it as "I understood how he thinks", it has succeeded.
