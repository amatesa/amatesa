# 14 — UX Blueprint

## Purpose

This document defines the complete user experience of the portfolio.

It does not describe implementation.

It describes what visitors should feel, discover and remember while exploring the website.

Every design, animation, interaction and component must support this experience.

---

# Experience Statement

This portfolio should not feel like reading a résumé.

It should feel like exploring the mind of someone who enjoys understanding complex systems and improving them.

Visitors should gradually discover who Alejandro is through interaction rather than through long explanations.

The experience should reward curiosity.

---

# The Journey

The experience follows five chapters.

Think

↓

Build

↓

Learn

↓

Explore

↓

Connect

Each chapter should naturally lead to the next.

The visitor should never feel lost.

---

# First Impression (0–5 seconds)

The visitor lands on the homepage.

The interface feels calm.

There is movement.

But nothing asks for attention.

The visitor notices that the page is alive.

Not animated.

Alive.

Nothing flashes.

Nothing rotates.

Nothing competes.

Curiosity appears before information.

---

# Recognition (5–20 seconds)

The visitor starts reading.

The headline is simple.

The interface remains quiet.

Small interactions suggest there is more to discover.

The visitor realizes this is not a traditional portfolio.

---

# Exploration (20–90 seconds)

Scrolling becomes exploration.

Sections are discovered rather than reached.

Every interaction reveals something meaningful.

Cards unfold.

Images expand.

Ideas connect.

Animations explain.

They never decorate.

---

# Understanding (90–180 seconds)

By now the visitor understands something important.

Alejandro is not presenting technologies.

He is presenting a way of thinking.

The projects become evidence.

Not the story itself.

---

# Decision (180+ seconds)

The visitor is no longer evaluating skills.

The visitor is imagining collaboration.

The desired internal question becomes:

"What would it be like working with this person?"

If the portfolio reaches this point, it has succeeded.

---

# Interaction Philosophy

Every interaction teaches something.

If an interaction does not communicate information, remove it.

Every click should reward curiosity.

Every hover should reinforce understanding.

Every animation should have intention.

---

# Scrolling

Scrolling is not transportation.

Scrolling is progression.

The user should feel that each section has been earned.

Transitions should gently prepare the next chapter.

---

# Motion

Movement has purpose.

Movement communicates hierarchy.

Movement communicates relationships.

Movement communicates causality.

Nothing moves because movement is possible.

---

# Time

The interface should never feel rushed.

Animations should feel responsive.

Not immediate.

Not slow.

Just enough to communicate intention.

---

# Navigation

The navigation remains available.

It never dominates.

It acts as orientation rather than control.

The visitor should always know where they are.

But never feel constrained.

---

# Discoverability

Every section should contain one small surprise.

Not a gimmick.

A reward.

Examples:

A project unfolding into a workspace.

A concept connecting to another.

A hidden explanation.

A smooth camera transition.

Micro-details that appear only after interaction.

---

# Reading

Text should breathe.

Large paragraphs should be avoided.

Whitespace is intentional.

The interface should encourage reading instead of demanding it.

---

# Visual Rhythm

Every chapter has a slightly different personality.

Think

Quiet.

Minimal.

Reflective.

---

Build

Constructive.

Structured.

Purposeful.

---

Learn

Connected.

Editorial.

Organized.

---

Explore

Experimental.

Playful.

---

Connect

Simple.

Human.

---

# Three.js Philosophy

Three.js is never used to impress.

Three.js exists to communicate ideas that HTML alone cannot express.

It should visualize relationships.

Movement.

Systems.

Connections.

Never decorative particles.

Never meaningless backgrounds.

Never random geometry.

---

# WebGL Rules

3D should feel integrated with the interface.

Not placed behind it.

Objects should behave like interface elements.

Not like game assets.

The user should always understand what is interactive.

---

# Camera Language

Camera movement replaces page transitions.

The camera never jumps.

It travels.

Every movement should feel motivated.

Like approaching an object to inspect it.

Or stepping back to understand a larger system.

---

# Interaction Language

Objects never explode.

Cards never flip dramatically.

Windows never bounce.

Everything behaves with physical consistency.

Objects slide.

Expand.

Align.

Connect.

Return.

Nothing teleports.

---

# Invisible Grid

Everything belongs somewhere.

Objects never float randomly.

Every movement ends in alignment.

Every transition respects the invisible structure of the page.

The user should never consciously notice this.

But they should feel it.

---

# Accessibility

Accessibility is part of the design.

Animations must respect reduced-motion preferences.

Every interaction must remain usable without animation.

Keyboard navigation must never be an afterthought.

Contrast is never sacrificed for aesthetics.

---

# Performance

Fast loading is part of the experience.

The page should progressively enhance itself.

Content appears first.

Effects appear second.

Interaction appears third.

The visitor should never wait for decoration.

---

# Mobile

Mobile is not a smaller desktop.

It is another experience.

Interactions should adapt naturally.

Three.js scenes should simplify.

Animations should become shorter.

Reading should become easier.

Nothing essential should disappear.

---

# Success Criteria

A successful visitor should leave thinking:

"I understand how this person thinks."

Not:

"I saw a cool website."

If visitors remember the interaction but forget Alejandro, the portfolio has failed.

If visitors remember Alejandro and barely notice the interface, the design has succeeded.

---

# Final Principle

The interface should quietly communicate the same idea as every document in this project.

Understanding comes first.

Everything else is built from there.
